# Стандарт приёмочного Allure-отчёта

Этот документ — единый критерий для ревьюеров (архитектор / безопасник / тестер).
Приёмка идёт **по Allure-отчёту**, поэтому отчёт обязан быть самодостаточным:
по нему, **не открывая код тестов**, должно быть понятно — ЧТО проверялось, КАК
и почему результат верный.

## Главный принцип

> Если по шагам отчёта нельзя восстановить сценарий и убедиться в корректности —
> отчёт **не принят**, даже если все тесты зелёные.

## Обязательные требования к отчёту

1. **Осмысленные имена тестов.** `@DisplayName` — человеческим языком, по делу
   («Создание заказа сохраняет его в БД и шлёт событие в Kafka»), а не имя метода.

2. **Структура шагов Given/When/Then (или Arrange/Act/Assert).** Каждое значимое
   действие и проверка — отдельный `Allure.step` с конкретным названием:
   что делаем и с какими данными (`When: POST /api/orders {productName=laptop, qty=2}`),
   а не `step1`, `выполнить`, `проверка`.

3. **Каждый значимый эффект — вложение (attachment) с понятным именем:**
   - HTTP — запрос и ответ (метод, путь, статус, тело);
   - БД — выполненный запрос/состояние сущности;
   - Kafka — отправленное/полученное сообщение (topic, key, value);
   - конфигурация — срез настроек;
   - логи приложения за время теста.

4. **Ассерты видны как шаги.** Из шага понятно «ожидали X = получили X»
   (а при падении — «ожидали X, получили Y»).

5. **Входные данные — в параметрах теста** (Allure parameters), чтобы было видно,
   на каких данных гонялся сценарий.

6. **Группировка проставлена:** `@Epic` / `@Feature` (и `@Story`, где уместно) —
   чтобы отчёт читался по фичам, а не плоским списком.

7. **Диагностируемость падений.** По упавшему шагу + вложению видно, ГДЕ и ПОЧЕМУ
   упало, без перезапуска и чтения исходников.

8. **Достоверность значений.** Показанное в шагах/вложениях обязано ПРАВДИВО
   отражать операцию: аргументы = то, что реально ОТПРАВИЛИ (а не мутированное
   состояние после вызова), результат = то, что вернулось. Значение, вводящее в
   заблуждение (классика: у `save` в аргументах виден сгенерированный БД `id`,
   которого на входе не было), — это ДЕФЕКТ, а не мелочь.

9. **Секреты (на текущем этапе).** Маскирование намеренно отключено — значения
   пишутся как есть, поэтому в тестах используем только фейковые/несекретные данные.
   Попавший в отчёт секрет — замечание информационное, не повод не принять.

## Чек-лист приёмки (быстрый)

- [ ] У каждого теста человекочитаемый `@DisplayName`.
- [ ] Видна структура Given/When/Then, шаги названы конкретно.
- [ ] Все внешние взаимодействия (HTTP/БД/Kafka/конфиг/логи) — во вложениях.
- [ ] Ассерты отражены шагами (ожидание vs факт).
- [ ] **Значения во вложениях достоверны** (прочитаны глазами, не по именам шагов):
      аргументы = отправленное, результат = вернувшееся, ничего не вводит в заблуждение.
- [ ] Проставлены Epic/Feature.
- [ ] В тестах только фейковые данные (маскирования секретов сейчас нет).
- [ ] По одному лишь отчёту можно провести приёмку сценария.

## Соглашение об именах вложений

Чтобы отчёт читался единообразно по всем модулям:
- **сетевые протоколы** (HTTP) → `HTTP Request` / `HTTP Response`;
- **вызовы методов** (репозиторий, мок, и т.п.) → `<Префикс> Call` / `<Префикс> Result`
  (напр. `DB Call` / `DB Result`).
Префикс шага — короткий маркер технологии (`HTTP`, `DB`, `Kafka`, `Mock`…).
Новые модули должны следовать этому словарю, а не плодить третий вариант.

## Исключение: внутренние тесты уровня A

Детерминированные тесты уровня A (через `InMemoryAllure`) проверяют содержимое
отчёта В ПАМЯТИ, поэтому в файловом отчёте (`allure-results`) они выглядят ПУСТЫМИ
(без шагов/вложений) — это by-design, не дефект. Человекочитаемая приёмка по отчёту
идёт по «живым» тестам `*ReportIT` (уровень B). При приёмке по отчёту смотри именно их.

## Как получить отчёт

```bash
cd ~/projects/allure-spring-test
mvn test                 # генерирует target/allure-results
mvn allure:report        # статический отчёт -> target/site/allure-maven-plugin
mvn allure:serve         # открыть в браузере
```

Для машинной проверки содержимого — читать JSON из `target/allure-results/`
(`*-result.json`: поля `name`, `steps`, `attachments`, `parameters`, `status`;
тексты вложений — в соседних `*-attachment.*`).

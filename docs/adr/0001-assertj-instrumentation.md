# ADR 0001 — Инструментирование AssertJ (самый хрупкий узел)

Статус: принято (2026-06-23). Короткая запись решения, чтобы человек при поддержке понял
«почему так» и не сломал при правке. Код: `assertion/AllureAssertJInstrumentation.java`.

## Контекст

Цель модуля — каждый AssertJ-ассерт в тесте даёт шаг «Проверка: значение X — method arg»
БЕЗ кода в тесте, включая КАСТОМНЫЕ ассерты потребителя (`extends AbstractAssert`). Сложность:
методы-проверки размазаны по ГЛУБОКОЙ иерархии абстрактных классов AssertJ
(`isEqualTo` — в `AbstractAssert`, `startsWith` — в `AbstractCharSequenceAssert`,
`contains`/`hasSize` — в `AbstractIterableAssert` и т.д.). Полнота перехвата зависит от того,
ВСЕ ли классы иерархии вплетены — а это зависит от ПОРЯДКА ЗАГРУЗКИ классов относительно
установки агента.

## Решение

1. **Blacklist, а не whitelist.** Вплетаем во ВСЕ public non-static методы любого подтипа
   `AbstractAssert`, КРОМЕ списка не-проверок (config/navigation/internal). Так ловим и
   кастомные ассерты с произвольными именами. При апгрейде AssertJ новые fluent/navigation-
   методы, возможно, придётся дописать в blacklist.
2. **Дедуп делегации счётчиком глубины** (`ThreadLocal`). `isEqualTo` делегирует в
   `super.isEqualTo` — оба вплетены; логируем только ВНЕШНИЙ (пользовательский) вызов
   (глубина 1), внутренние делегаты пропускаем.
3. **Установка через `AgentBuilder…DiscoveryStrategy.Reiterating`.** Дефолтный батч
   `RETRANSFORMATION` пропускает УЖЕ загруженные классы; абстрактные предки AssertJ
   Spring/surefire подтягивают РАНО (до агента) → без Reiterating их методы остаются без
   advice и строковые/коллекционные ассерты молча выпадают из отчёта (это был реальный баг).
   Reiterating пересканирует загруженные классы, пока набор не стабилизируется.

## Осознанные компромиссы

- **Листовые конкретные ассерты (`StringAssert` и т.п.) при вплетении ОШИБАЮТСЯ** — у них
  ПУБЛИЧНЫЙ конструктор, а advice с `onThrowable` нельзя положить на конструктор
  («Cannot catch exception during constructor call»). Это ТЕРПИМ: их методы-проверки
  наследуются из абстрактных классов, которые вплетены успешно. Матчер НЕ усложняем.

## Отвергнутые альтернативы (НЕ возвращать — проверено, ломается)

- **Исключить конструкторы/synthetic из матчера** (`not(isConstructor())`/`not(isSynthetic())`),
  чтобы листовые классы тоже вплетались. → Меняет НАБОР вплетённых классов и
  РАССИНХРОНИВАЕТ дедуп по глубине: то пропадают comparable-ассерты (`isBetween`), то
  коллекционные. Каждая правка просто перемещает поломку. Откатано.
- **Ручной `instrumentation.retransformClasses(...)`** для уже загруженных классов. →
  Конфликтует с future-load перехватом AgentBuilder: классы, загружаемые ПОЗЖЕ, перестают
  вплетаться. Откатано в пользу Reiterating.

## Хрупкость и что делать при апгрейде AssertJ

- Дедуп ЧУВСТВИТЕЛЕН к набору вплетённых классов — меняя матчер, жди рассинхрона.
- Прогнать после апгрейда/правки:
  - `e2e/AssertJReportIT` — end-to-end полнота (строковые/коллекционные/comparable шаги есть);
  - `assertion/AllureAssertJTest#successfulAssertSingleStep` — дедуп (ровно один шаг);
  - `internal/InstrumentationApiCanaryTest#assertjHierarchy` — иерархия/методы на местах.
- Диагностика хрупкого байткода: повесить `AgentBuilder.Listener` (лог transform/ignore/error
  по типам) в `internal/AllureInstrumentation`; для дедупа — временная трасса enter/exit в
  advice. См. грабли в `docs/java-code-standard.md`.

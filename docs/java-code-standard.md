# Стандарт качества кода (Java)

Этот документ — единый кодекс качества **кода** библиотеки `allure-spring-test`.
На него опирается ревьюер `java-lead` (лидер компетенции по Java). Отдельно от
`docs/acceptance-report-standard.md` (тот — про качество Allure-**отчёта**): отчёт
принимает tester по нему, код принимает java-lead по этому.

Библиотека — **продукт** для чужих команд. Принцип: код обязан быть образцовым —
идиоматичным, потокобезопасным, без утечек, понятным без автора.

## Главный принцип

> Если поведение кода нельзя понять, не запуская его, или оно ломается в параллельных
> тестах / течёт между ними — код **не принят**, даже если тесты зелёные.

## 1. Идиоматичная Java 21

- records — для неизменяемых носителей данных; sealed — для закрытых иерархий;
  switch-patterns вместо лестниц `instanceof`.
- `var` — где тип очевиден справа и не теряется читаемость.
- Стримы там, где они читаемее цикла; не ради моды (горячий путь инструментирования —
  обычный цикл уместнее).
- `Optional` — как возвращаемое значение, НЕ как поле или параметр метода.
- Иммутабельность и `final` по умолчанию; никаких сырых типов и лишних боксингов.
- Современно, но без переусложнения: простое решение бьёт «умное».

## 2. Потокобезопасность (главное в этой библиотеке)

Инструментирование работает в параллельных тестах — это зона повышенного риска.

- ByteBuddy-`retransform`/агенты ставятся один раз, идемпотентно; гонок установки нет.
- `ThreadLocal`-гарды (Spring-ассерты) и счётчики глубины (AssertJ) — снимаются в
  `finally`, не текут между тестами; декремент парный инкременту.
- Общее мутабельное состояние — под синхронизацией или `volatile`/atomics; видимость
  (happens-before) обеспечена.
- Код обязан быть корректен под `@Execution(CONCURRENT)`; глобальный Allure lifecycle
  не путается между потоками.
- Кастомный `MockMaker` потокобезопасен и не конфликтует с пользовательским.

## 3. Ресурсы и жизненный цикл

- try-with-resources на всё закрываемое.
- Logback-аппендеры снимаются после теста; `retransform`/инструментирование
  откатывается; ничего не остаётся «висеть».
- Глобальный Allure lifecycle восстанавливается через `InMemoryAllure.uninstall` —
  вернуть ИМЕННО прежний экземпляр, НЕ `new AllureLifecycle()` (allure-junit кэширует
  ссылку → «no test case running» и потеря вложений в живых тестах).
- Между тестами не течёт состояние (фикстуры чистятся, статика сбрасывается).

## 4. Обработка исключений и null-safety

- Не глотать исключения молча; нет `catch (Exception ignored)`.
- Падения САМОГО инструментирования видны (`onThrowable`/`@Thrown`), а не маскируют
  результат теста и не роняют чужой код.
- **Не переизобретать фреймворк.** Падение пользовательской проверки/операции показывает
  Allure из коробки (тест failed + сообщение/стек в результате теста). НЕ фабриковать
  руками FAILED/BROKEN-шаги и НЕ дублировать текст исключения в шаг/вложение/`statusDetails`.
  Инструментируем УСПЕШНЫЙ путь (шаг для прошедшей операции); упавший — отдаём Allure.
  Если ради «статуса упавшего шага» появляется ручной `startStep/stopStep`+`setStatus` —
  это запах: скорее всего, шаг тут вообще не нужен.
- Контрактная null-семантика параметров/возвратов понятна; без NPE-ловушек.

## 5. Эргономика API и упаковка

- Имена пакетов/классов/свойств — в стиле экосистемы Allure; публичный контракт
  минимален, внутренности скрыты (`private`/package-private по умолчанию).
- Нет дублирования: общая база — `internal/AllureInstrumentation`, а не копипаста.
- Лямбды `Allure.step(name, ...)` — БЛОЧНЫЕ `{ ... }` (иначе неоднозначность
  void/возврат, перегрузки конфликтуют).
- feature-зависимости — scope `provided`/`optional` (есть у потребителя в test-scope);
  `compile` — только своё API. Без лишних транзитивов и fat-jar.

## 6. Javadoc

- Публичный API задокументирован: что делает, какие тумблеры
  (`allure.spring.*.enabled` и пр.), потокобезопасность, side-effects активации.
- Без пустого javadoc «возвращает результат» — только по делу.

## 7. Качество тестового кода

Тестовый код — тоже код продукта.

- Без копипасты; переиспользует support-фикстуры (`InMemoryAllure`, `*TestApp`,
  `TestContexts`).
- Детерминизм на уровне кода (не зависит от порядка/времени/сети без нужды).
- ⚠️ НЕТ ручных `Allure.step` ради красоты отчёта — степы только от инструментирования
  (это и дефект отчёта у tester, и дефект кода здесь: подделка + лишний код в тесте).

## Чек-лист код-ревью (быстрый)

- [ ] Идиомы Java 21 к месту, без переусложнения; `final`/иммутабельность по умолчанию.
- [ ] Потокобезопасно: нет гонок, видимость обеспечена, `ThreadLocal`/счётчики
      снимаются в `finally`, корректно под параллельным запуском.
- [ ] Ресурсы закрываются; аппендеры/инструментирование откатываются; lifecycle
      восстановлен через `uninstall` (прежний экземпляр).
- [ ] Исключения не проглочены; падения инструментирования видны.
- [ ] Публичный API минимален и задокументирован javadoc; внутренности скрыты.
- [ ] Нет дублирования (общая база `internal/`); лямбды `Allure.step` блочные.
- [ ] feature-зависимости в правильном scope (`provided`/`optional`).
- [ ] Тестовый код без копипасты, переиспользует фикстуры; нет ручных `Allure.step`.

## Грабли (уроки проекта — проверять всегда)

1. **Лямбда `Allure.step(name, ...)` должна быть блочной** — иначе неоднозначность
   void/возврат, конфликт перегрузок.
2. **Дубли от делегации гасить:** thread-local guard (Spring-ассерты), счётчик глубины
   (AssertJ), `matchArguments(2)` (Kafka producer) — иначе один эффект дублируется.
3. **SPI-файл MockMaker — только в `src/test/resources`** (не в jar), иначе навяжется
   всем потребителям и сломает их MockMaker. Потребитель добавляет файл сам (README).
4. **Восстановление lifecycle — через `InMemoryAllure.uninstall`**, не `new
   AllureLifecycle()` (см. §3).
5. **Сбой САМОГО инструментирования** (ByteBuddy/advice) логируется на WARNING
   (`AllureInstrumentationLogger`), а не молча и НЕ фабрикованным FAILED-шагом. Упавшую
   пользовательскую проверку отдаём Allure (см. §4) — шаг для неё не создаём.

## Как смотреть код

```bash
cd ~/projects/allure-spring-test
# JDK рулит jenv (дефолт 21); в фоновом шелле звать через zsh -lic
zsh -lic 'mvn -q -DskipTests compile'   # warning'и компилятора
zsh -lic 'mvn test'                     # полный прогон
```

Читать `src/main` и `src/test` целиком вокруг изменений, включая точки взаимодействия
(общая инструментация `internal/`, гарды). Не доверять описанию — проверять в коде.
